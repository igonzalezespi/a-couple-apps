# Foundation + movies app review (2026-05)

A multi-agent review of the whole project after the foundation and the movies app were
built (PR #1). Covers: are the checks truly done, what could be better, the critical path
to a real v1, and how to make the app genuinely useful for a couple. This is a snapshot for
review -- reconcile the OpenSpec task checkboxes via `/opsx:verify` (not hand-edited here).

## Verdict

The engineering is real and good on the **web** target and the data layer: the gate
(lint/typecheck/87+ tests/build) passes, the Playwright web smoke runs green, the Supabase
schema + RLS + realtime are sound, and the web app renders correctly on a real Android
emulator (light + dark). The **process scaffolding (CI, the OpenSpec verify->archive loop)
is written but never executed**, **native (iOS/Android) has never been built or run**, and
the app currently **stores** a watchlist but does not yet help a couple **decide** what to
watch -- so it is functional, not yet loved.

## Could it be done better?
- **Prove native once** (bundle ids + `eas.json` + one EAS/local build + run Maestro) -- it is the
  only Phase-6 acceptance with zero evidence and the riskiest integration (Tamagui + Metro + RN).
- **Commit `supabase/config.toml`** (expose `shared`+`movies`) + a `pnpm db:types` script, so the
  API exposure and DB types are reproducible, not maintainer-local prose.
- **Split PR #1** into two (foundation, then movies) to match the two OpenSpec changes and make
  review tractable; dogfood the `/opsx:verify`+`/opsx:archive` loop so `openspec/specs/` becomes real.
- **Pay the Phase-7 prerequisite before app #2**: per-app schema types are hard-coded (and dead)
  in `packages/core/src/types.ts`, and i18n uses one flat namespace -- both force edits to shared
  packages when `apps/plans` lands. Make `Database` composable per-app and add per-app i18n namespaces.

## Critical path to a usable v1 + release
1. Push CI (`gh auth refresh -h github.com -s workflow`) -> let `ci-gate` run once.
2. Maintainer 9.3 two-tab realtime check (or set up custom SMTP first).
3. Merge PR #1 (ideally split in two); run `/opsx:verify` + `/opsx:archive` to close the SDD loop.
4. Prove a native build: set `ios.bundleIdentifier`/`android.package`, add `eas.json`, build once, run Maestro.
5. Commit `supabase/config.toml`; add `pnpm db:types`.
6. Phase-7 prerequisite refactor (core schema types + i18n namespaces).
7. Make the watchlist couple-friendly (see below) -> this is what makes it a real v1.
8. Release (Phase 10): add `release.yml` + `renovate.json`, verify the fresh-clone check, tag `v0.1.0`.

## Make it useful + friendly (product)
Shipped this round: to-watch / watched split, "added by you / your partner" attribution,
empty-state call to action, de-emphasized language toggle.

Top remaining, by value:
1. **"Tonight's pick" nomination** (the one big idea): either partner floats a movie to the top as
   tonight's pick; the other sees it. A coordination primitive that turns a list into a decision.
   Needs a small schema change (`picked_by`/`picked_at`) + realtime + one UI treatment.
2. **Movie detail view** (overview, rating, year) so a couple can choose from the list.
3. **Optimistic watched toggle + undo on remove** (instant feedback; safe shared deletes).
4. **Resend-code + clearer auth errors** (OTP expiry recovery).
5. **Search-as-you-type (debounce) + clear button**; **real settings screen** for language.
